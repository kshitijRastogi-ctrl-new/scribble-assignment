# Implementation Plan: Game Start & Drawer Flow

**Branch**: `002-game-start-drawer` | **Date**: 2026-05-31 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/002-game-start-drawer/spec.md`

---

## Summary

Extends the Scenario 1 `startGame` backend function and `GamePage` frontend to assign
drawer/guesser roles, select a secret word deterministically, filter it from the API
response for non-drawers, and render role-specific UI on the Game screen with 2-second
polling. All changes are brownfield extensions to existing files; no new npm dependencies
are introduced.

---

## 1. Data Model Changes

### 1a. `backend/src/models/game.ts`

**`RoomStatus`** — add `"result"` (FR-012 navigation trigger; set by Scenario 3):
```
"lobby" | "playing"   →   "lobby" | "playing" | "result"
```

**`Participant`** — add `role` field. Typed as `string` to accommodate the `""` default
(unset in lobby). The existing `ParticipantRole` type (`"drawer" | "guesser"`) remains
the canonical union for assigned values and is already in this file:
```
Before: { id, name, isHost, score, joinedAt }
After:  { id, name, isHost, score, joinedAt, role: string }
```

> **Type note (G1)**: `role` is typed as `string` in the interface to avoid adding a
> fourth value to `ParticipantRole`. In practice the only values ever written are
> `"drawer"`, `"guesser"`, and `""` (empty string = unassigned / lobby state). Tasks
> will document this explicitly.

**`Room`** — add two new fields (Q2):
```
Before: { code, host, status, participants, createdAt, updatedAt }
After:  { code, host, status, participants, createdAt, updatedAt,
          wordIndex: number, secretWord: string }
```
Both initialise to `0` and `""` respectively in `createRoom`.

**`RoomSnapshot`** — remove `roles: ParticipantRole[]` (Q1), add `secretWord?` (Q5):
```
Before: { code, host, status, participants, availableWords, roles: ParticipantRole[] }
After:  { code, host, status, participants, availableWords, secretWord?: string }
```
`participants` entries now carry `role: string` transitively via the updated `Participant`.

### 1b. `frontend/src/services/api.ts`

Mirror of backend model changes (Q1, Q5):

- `Participant`: add `role: string`
- `RoomSnapshot`: remove `roles: ParticipantRole[]`, add `secretWord?: string`, extend `status` to `"lobby" | "playing" | "result"`
- `ParticipantRole` type and its usage: removed (only needed backend-side; frontend no longer has `roles[]`)

All changes extend the existing function bodies. No function is moved or replaced.

**`createParticipant(name, isHost)`** — add `role: ""` to returned object literal:

```

**`createRoom(playerName)`** — add `wordIndex: 0` and `secretWord: ""` to room literal:
```

**`toRoomSnapshot(room, viewerName?: string)`** — rename param (Q3), implement
secretWord filtering (FR-003/FR-004), remove `roles[]` output (Q1):
```
Rename:  viewerParticipantId → viewerName

Logic:
  const isDrawer = !!viewerName &&
    room.participants.some(p => p.name === viewerName && p.role === "drawer");

Return shape:
  {
    code, host, status,
    participants: room.participants.map(p => ({ ...p })),  // role now on each p
    availableWords: listWords(),
    ...(isDrawer && room.status === "playing" ? { secretWord: room.secretWord } : {})
    // roles[] dropped entirely
  }
```

> **Cleanup note (G2)**: Remove the `roles: [...STARTER_ROLES]` line from the returned
> object. The `STARTER_ROLES` import in `roomStore.ts` becomes unused after this removal
> and must be deleted from the import statement at the top of the file.

**`startGame(code, playerName)`** — extend with guard + role assignment + word selection
(FR-001, FR-002, FR-014). Insertion points within the existing function body:

```
Existing flow:          Scenario 2 additions (bold markers show insertion points):
  null guard            (unchanged)
  notHost guard         (unchanged)
  notEnoughPlayers      (unchanged)
  ► INSERT: alreadyStarted guard
    if (room.status === "playing") return "alreadyStarted" as const;
  room.status = "playing"   (unchanged)
  ► INSERT: role assignment
    room.participants.forEach(p => { p.role = p.isHost ? "drawer" : "guesser"; });
  ► INSERT: word selection
    room.secretWord = STARTER_WORDS[room.wordIndex % STARTER_WORDS.length];
    room.wordIndex += 1;
  saveRoom(room)        (unchanged)
  return toRoomSnapshot(room)  (unchanged — but toRoomSnapshot now accepts no viewerName
                                here, so secretWord is NOT in the start response itself)
```

> **Response note (G3)**: `POST /rooms/:code/start` never includes `secretWord` in its
> response. The drawer receives `secretWord` on the first `GET /rooms/:code?player=<name>`
> poll response. This is intentional and consistent with FR-003/FR-004 role-based
> filtering — the start response has no `viewerName` context, so the guesser-safe
> (no-secretWord) shape is always returned.

```
```

Updated return type:
```typescript
null | "notHost" | "notEnoughPlayers" | "alreadyStarted" | RoomSnapshot
```

Note: `toRoomSnapshot(room)` is called with no `viewerName` at the end of `startGame`,
so the start response never includes `secretWord`. The drawer fetches their word via
the first poll (GET with `?player=<name>`).

## 4. API Contract

### GET /rooms/:code

Accepts optional `?player=<name>` query parameter (replaces unused `?participantId=`).

| Caller / status | `secretWord` in response? | `role` on participants? |
|----------------|--------------------------|------------------------|
| Any / `"lobby"` | No (not yet set) | `""` on all |
| Drawer / `"playing"` | **Yes** | `"drawer"` / `"guesser"` |
| Guesser / `"playing"` | No | `"drawer"` / `"guesser"` |
| No param / `"playing"` | No | `"drawer"` / `"guesser"` |
| Unknown name / `"playing"` | No | `"drawer"` / `"guesser"` |

**Drawer response** (`?player=Alice`, Alice is drawer, status `"playing"`):
```json
{
  "room": {
    "code": "ABCD",
    "host": "Alice",
    "status": "playing",
    "secretWord": "rocket",
    "availableWords": ["rocket","pizza","castle","guitar","sunflower"],
    "participants": [
      {"id":"…","name":"Alice","isHost":true,"role":"drawer","score":0,"joinedAt":"…"},
      {"id":"…","name":"Bob","isHost":false,"role":"guesser","score":0,"joinedAt":"…"}
    ]
  }
}
```

**Guesser / no-param response** (Bob, or no `?player=`):
```json
{
  "room": {
    "code": "ABCD",
    "host": "Alice",
    "status": "playing",
    "availableWords": ["rocket","pizza","castle","guitar","sunflower"],
    "participants": [
      {"id":"…","name":"Alice","isHost":true,"role":"drawer","score":0,"joinedAt":"…"},
      {"id":"…","name":"Bob","isHost":false,"role":"guesser","score":0,"joinedAt":"…"}
    ]
  }
}
```

| Condition | HTTP Status | Body |
|-----------|------------|------|
| `room.status === "playing"` already | 400 | `{ "error": "Game already started" }` |

## Complexity Tracking

| Decision | Alternatives considered | Chosen rationale |
|----------|------------------------|-----------------|
| `fetchRoomByCode` used for BOTH mount AND polling | Use `fetchRoom` after mount sets `state.room` | No timing dependency between effects; works reliably after page refresh |
| `secretWord` filtered in `toRoomSnapshot` via spread | Separate snapshot builder for drawer vs guesser | Single function, minimal diff, brownfield-compatible |
| `role: string` on `Participant` (not `ParticipantRole \| ""`) | Use explicit three-value union | Avoids adding `""` to `ParticipantRole`; empty string is the unset default, not an assigned role |
| `ResultPage.tsx` as minimal new file | Inline element in routes | Keeps routing pattern consistent; prevents wildcard redirect consuming FR-012 navigation |
| `startGame` returns `"alreadyStarted"` discriminant | Throw inline in service | Consistent with existing discriminant pattern (`"notHost"`, `"notEnoughPlayers"`) |
