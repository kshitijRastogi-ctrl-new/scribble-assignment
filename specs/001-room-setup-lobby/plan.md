# Implementation Plan: Room Setup & Lobby

**Branch**: `001-room-setup-lobby` | **Date**: 2026-05-30
**Spec**: [spec.md](./spec.md)

## Summary

Extend the Scribble starter (which has room create/join and a manual-refresh lobby) to add:
host tracking, player name validation, room code validation, duplicate name rejection,
auto-polling, and a host-gated start-game flow. All changes are additive — no existing
working behavior is removed.

## Technical Context

**Language/Version**: TypeScript (strict) — Node.js 18+ backend, React 18 frontend

**Primary Dependencies**: Express (backend), React Router v6 (frontend), Zod (validation) — all already present

**Storage**: In-memory `Map<string, Room>` in `backend/src/services/roomStore.ts` — no DB

**Testing**: Existing test files present (`schemas.test.ts`, `roomStore.test.ts`, `api.test.ts`)

**Target Platform**: Local dev — `localhost:3001` (backend), `localhost:5173` (frontend)

**Project Type**: Brownfield web application enhancement

**Performance Goals**: Polling latency ≤ 2s for participant list updates

**Constraints**: No WebSockets, no database, no auth tokens, no new top-level dependencies

**Scale/Scope**: 2–10 players per room, single room per browser session

## Data Flow

### Create Room Flow

```
CreateRoomPage
  → trim name, validate non-empty (client)
  → POST /rooms { playerName }
  → backend: Zod trim+min(1), createRoom()
      → Participant { isHost: true, score: 0 }
      → Room { host: name, status: "lobby" }
  → response 201 { participantId, room }
  → roomStore.setRoomSession()
  → localStorage.setItem("playerName", name)
  → localStorage.setItem("roomCode", code)
  → navigate("/lobby")
```

### Join Room Flow

```
JoinRoomPage
  → trim name + code, validate non-empty (client)
  → POST /rooms/:code/join { playerName }
  → backend: Zod trim+min(1), code.trim().toUpperCase()
      → check room exists (404 if not)
      → check name not duplicate (409 if taken)
      → Participant { isHost: false, score: 0 }
  → response 200 { participantId, room }
  → roomStore.setRoomSession()
  → localStorage.setItem("playerName", name)
  → localStorage.setItem("roomCode", code)
  → navigate("/lobby")
```

### Lobby Polling Flow

```
LobbyPage mount
  → read playerName from localStorage
  → setInterval(poll, 2000)
      poll():
        GET /rooms/:code
        → update roomStore snapshot
        → if status === "playing": navigate("/game/${room.code}")
  → unmount → clearInterval (useEffect cleanup)
```

### Start Game Flow

```
Host clicks "Start Game"
  → POST /rooms/:code/start { playerName }
  → backend: check playerName === room.host (403 if not)
  → backend: check participants.length >= 2 (400 if not)
  → room.status = "playing"
  → response 200 { room }
  → next poll by all clients detects "playing" → navigate("/game/${room.code}")
```

## Backend File Changes

### `backend/src/models/game.ts`

- `RoomStatus`: add `"playing"` alongside existing `"lobby"` — type becomes `"lobby" | "playing"` (no rename; brownfield-first)
- `Participant`: add `isHost: boolean`, `score: number`
- `Room`: add `host: string`
- `RoomSnapshot`: add `host: string`; update `status` type; participants now include `isHost` + `score`

### `backend/src/api/schemas.ts`

- `createRoomSchema.playerName`: `z.string().optional()` → `z.string().trim().min(1, { message: "Name cannot be empty" })`
- `joinRoomSchema.playerName`: same change
- Add `startGameSchema`: `z.object({ playerName: z.string().trim().min(1) })`

### `backend/src/services/roomStore.ts`

- Remove `displayName()` fallback helper
- `createParticipant(name, isHost)`: add `isHost` param, add `score: 0`
- `createRoom(playerName)`: creator is `isHost: true`; set `room.host = playerName`
- `joinRoom(code, playerName)`:
  - Validate code is non-empty (return discriminant `"emptyCode"`)
  - Check duplicate name after trim (return discriminant `"duplicateName"`)
  - New participant is `isHost: false`
- Add `startGame(code, playerName)`:
  - Returns `null` if room not found
  - Returns `"notHost"` string if `playerName !== room.host`
  - Returns `"notEnoughPlayers"` string if `participants.length < 2`
  - Sets `room.status = "playing"`, returns updated room snapshot
- `toRoomSnapshot()`: include `host` field from room

### `backend/src/api/rooms.ts`

- `POST /rooms`: Zod failure now triggers 400 with `{ error: "Name cannot be empty" }` via errorHandler
- `POST /rooms/:code/join`:
  - Check `code.trim()` empty → `HttpError(400, "Room code cannot be empty")`
  - Map `joinRoom` discriminants → 404 `"Room not found"` or 409 `"Name already taken in this room"`
- `GET /rooms/:code`: change error message from `"Unable to load room"` → `"Room not found"`
- Add `POST /rooms/:code/start`: parse `startGameSchema`, call `startGame()`, map discriminants to 403/400/404 responses

### `backend/src/api/router.ts`

- `errorHandler`: `{ message }` → `{ error }` in all response paths
- Zod branch: `{ message: "Invalid request payload" }` → `{ error: "Invalid request payload" }`

## Implementation Sequence

Implement in this order — each step is independently verifiable:

1. `backend/src/models/game.ts` — types only, no runtime impact; TypeScript catches downstream mismatches immediately
2. `backend/src/api/schemas.ts` — Zod schemas tighten; empty names now rejected at schema layer
3. `backend/src/services/roomStore.ts` — core logic: host tracking, duplicate check, startGame
4. `backend/src/api/rooms.ts` — wire up new logic and error messages
5. `backend/src/api/router.ts` — fix error key globally
6. `frontend/src/services/api.ts` — types + new method + URL fix + error key
7. `frontend/src/state/roomStore.ts` — localStorage writes + startGame method
8. `frontend/src/pages/CreateRoomPage.tsx` + `JoinRoomPage.tsx` — client-side validation
9. `frontend/src/routes/index.tsx` — route param change
10. `frontend/src/pages/LobbyPage.tsx` — polling + host UI
11. `frontend/src/pages/GamePage.tsx` — useParams

## Testing Strategy

Manual validation with two browser tabs after each step. Existing test files
(`schemas.test.ts`, `roomStore.test.ts`, `api.test.ts`) updated alongside each
implementation step to reflect changed error key and new fields.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Error key change (`message` → `error`) breaks test assertions | Update test expectations in same commit as handler change |
| `fetchRoom()` in polling has no debounce — could queue if server is slow | Acceptable for local dev; 2s interval is generous |
| `playerName` in localStorage could be stale after visiting a different room | Single-room-per-tab assumption; multi-room juggling is out of scope |
