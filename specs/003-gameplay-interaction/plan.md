# Implementation Plan: Gameplay Interaction

**Branch**: `003-gameplay-interaction` | **Date**: 2026-05-31 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/003-gameplay-interaction/spec.md`

---

## Summary

Extends the Scenario 2 `GamePage` and backend `roomStore` to add interactive drawing,
guess submission, scoring, and round-end signalling. Three main loops are added:

1. **Draw → Upload → Poll**: Drawer draws on a real HTML canvas; uploads state on each
   stroke end via `POST /rooms/:code/canvas`; guessers receive it on their 2s poll.
2. **Guess → Score → Status**: Guessers submit trimmed text via `POST /rooms/:code/guess`;
   backend scores and appends to history; first correct guess transitions room to `"result"`.
3. **Display**: `Scoreboard` and `ResultPanel` components are wired to live polling data;
   `GamePage` polling (already in place from Scenario 2) navigates to `/result` on
   `"result"` status.

All changes are brownfield extensions. One new component (`DrawingCanvas`) is added to
encapsulate canvas interaction — justified by the complexity of the mouse event handling
and the need to keep `GamePage` readable. No new npm dependencies are introduced.

## 2. Backend Changes

### 2a. `backend/src/api/schemas.ts`

Two new Zod schemas appended at the end of the file:

```
canvasSchema = z.object({ canvasData: z.string() })

guessSchema = z.object({
  playerName: z.string().trim().min(1, { message: "Name cannot be empty" }),
  guess: z.string()
})
```

### 2b. `backend/src/services/roomStore.ts`

**`createRoom`** — add two fields to the room literal:
```
canvasData: "",
guesses: []
```

**`toRoomSnapshot`** — add to returned object (after `availableWords`):
```
canvasData: room.canvasData,
guesses: room.guesses.map(g => ({ ...g })),
```
The `secretWord` conditional spread remains unchanged at the end.

**New function `updateCanvas`** (add after `startGame`):
```
Signature: updateCanvas(code: string, canvasData: string): RoomSnapshot | null

1. rooms.get(code) → if not found, return null
2. room.canvasData = canvasData
3. saveRoom(room)
4. return toRoomSnapshot(room)   ← no viewerName needed; canvasData always returned
```

**New function `submitGuess`** (add after `updateCanvas`):
```
Signature: submitGuess(
  code: string,
  playerName: string,
  guess: string
): RoomSnapshot | null | "roundOver"

1. rooms.get(code) → if not found, return null
2. if room.status === "result" → return "roundOver" as const
3. const trimmed = guess.trim()   (defence-in-depth; frontend already trims)
4. const isCorrect = trimmed.toLowerCase() === room.secretWord.toLowerCase()
5. if isCorrect:
     find participant by name → if found: participant.score += 100
     room.status = "result"
     // INVARIANT: do not clear guesses or canvasData — Scenario 4 result screen depends on both
6. room.guesses.push({ playerName, text: trimmed, isCorrect })
7. saveRoom(room)
8. return toRoomSnapshot(room)
```

> **G1**: `saveRoom` already sets `updatedAt` and deep-clones before storing, so all
> mutations (score, status, guesses) are persisted atomically in step 7.
>
> **G2**: If `playerName` is not found in participants (late joiner, role `""`), the
> guess is still appended; the score increment is silently skipped. This is safe
> because only named participants with `role === "guesser"` can reach this endpoint
> via the GamePage UI (Scenario 2 gating).

### 2c. `backend/src/api/rooms.ts`

**Import changes** — extend existing destructure lists (no new `import` lines):
- From schemas: add `canvasSchema`, `guessSchema`
- From roomStore: add `updateCanvas`, `submitGuess`

**New `POST /:code/canvas` handler** (insert before `return router`):
```
1. safeParse code from params with roomCodeParamsSchema.safeParse(request.params) → 400 on failure
   (use safeParse, not parse/throw — consistent with POST /:code/guess handler)
2. safeParse body with canvasSchema → 400 "Invalid canvas data" on failure
3. call updateCanvas(code.toUpperCase(), parsed.data.canvasData)
4. if null → 404 "Room not found"
5. response.json({ ok: true })
```

**New `POST /:code/guess` handler** (insert after `/:code/canvas`):
```
1. safeParse code from params → 400 on failure
2. safeParse body with guessSchema → 400 with Zod message on failure
3. call submitGuess(code.toUpperCase(), playerName, guess)
4. if null → 404 "Room not found"
5. if "roundOver" → 400 "Round already over"
6. response.json({ room: result })
```

### 3e. `frontend/src/components/DrawingCanvas.tsx` — NEW FILE

**Purpose**: Encapsulates HTML canvas interaction. Keeps `GamePage` free of refs, mouse
handlers, and `drawImage` logic.

**Props**:
```typescript
interface DrawingCanvasProps {
  role: string;                             // "drawer" | "guesser" | ""
  canvasData: string;                       // from polling; "" = blank
  onStrokeEnd: (dataUrl: string) => void;  // fired on mouseup, mouseleave, and clear
}
```

**Internal refs/state**:
```typescript
canvasRef: useRef<HTMLCanvasElement>(null)
isDrawing: useState(false)
```

**Canvas dimensions**: `800 × 500` px (fixed; matches existing placeholder height).

**Mouse handlers** — only active when `role === "drawer"`:
```
onMouseDown:  isDrawing = true; ctx.beginPath(); ctx.moveTo(x, y)
onMouseMove:  if isDrawing: ctx.lineTo(x, y); ctx.stroke()
              (lineWidth: 4, strokeStyle: "#000000", lineCap: "round")
onMouseUp:    if isDrawing: isDrawing = false; onStrokeEnd(canvas.toDataURL("image/png"))
onMouseLeave: same as onMouseUp (stroke ends when cursor leaves canvas)
```

**`useEffect` — render incoming `canvasData`** (deps: `[canvasData]`):
```
if canvasData === "":
  ctx.clearRect(0, 0, canvas.width, canvas.height)
  return
const img = new Image()
img.onload = () => { ctx.clearRect(…); ctx.drawImage(img, 0, 0) }
img.src = canvasData
```

> **G3**: This effect fires for both drawer and guesser. For the drawer it enables
> canvas recovery after page refresh (the first poll populates `canvasData` with the
> last uploaded state). For guessers it keeps the display in sync with polling.
>
> **Known limitation**: polling every 2s may cause canvas flicker if a guesser's poll
> arrives mid-stroke (the `drawImage` call briefly overwrites in-progress local drawing).
> This is acceptable for lab scope and does not violate any spec requirement.

**Clear button** (rendered only when `role === "drawer"`):
```
onClick:
  ctx.clearRect(0, 0, canvas.width, canvas.height)
  onStrokeEnd(canvas.toDataURL("image/png"))   ← uploads blank canvas to backend
```

**Render**:
```tsx
<div className="drawing-canvas-wrapper">
  <canvas ref={canvasRef} width={800} height={500}
    style={{ border, backgroundColor: "#ffffff",
             cursor: role === "drawer" ? "crosshair" : "default" }}
    onMouseDown={…} onMouseMove={…} onMouseUp={…} onMouseLeave={…}
  />
  {role === "drawer" && (
    <div className="button-row button-row--compact">
      <button className="button button--secondary" onClick={handleClear}>Clear</button>
    </div>
  )}
</div>
```

### 3f. `frontend/src/pages/GamePage.tsx`

**New import**: `DrawingCanvas` from `../components/DrawingCanvas`

**Import `api`**: `GamePage.tsx` currently does not import `api` directly — it only
uses `roomStore` methods. Add `import { api } from "../services/api"` to the existing
import block. This is needed for `handleCanvasUpdate` and `handleGuessSubmit` to call

**New callbacks** (add before `return`):
```typescript
async function handleCanvasUpdate(dataUrl: string) {
  if (!code) return;
  api.updateCanvas(code, dataUrl).catch(err =>
    setRefreshError(err instanceof Error ? err.message : "Canvas upload failed"));
}

async function handleGuessSubmit(guess: string) {
  if (!code || !playerName) return;
  const result = await api.submitGuess(code, playerName, guess);
  roomStore.setRoomSnapshot(result.room);
  if (result.room.status === "result") navigate("/result");
}
```

> **G4**: `handleGuessSubmit` calls `roomStore.setRoomSnapshot` immediately so the
> guesser who submitted sees the updated score and history without waiting for the next
> 2s poll. The existing `polling useEffect` handles the same transition for all other
> players within one poll cycle.

## 6. Complexity Tracking

| Decision | Alternatives considered | Chosen rationale |
|----------|------------------------|-----------------|
| New `DrawingCanvas` component vs inline in `GamePage` | Inline: fewer files; Component: clean separation | Component — canvas needs `useRef`, 4 mouse handlers, a `useEffect` for incoming data, and a clear button; inlining adds 80+ lines to `GamePage` |
| Canvas upload on stroke end only vs throttled mousemove | Per-mousemove: smoother; throttled: complex; stroke-end: simple | Stroke-end aligns with REST-polling-only constraint; guessers poll 2s so intermediate frames are never displayed |
| `canvasData` returned to all vs guessers only | All: simpler; Guessers-only: mirrors `secretWord` | All — no security concern; avoids second conditional in `toRoomSnapshot` |
| Immediate `setRoomSnapshot` after guess vs wait for poll | Wait: simpler; Immediate: better UX | Immediate — the guesser who scores should see the updated score immediately; round-end navigation must not lag 2s |
| Backend trim of `guess` (defence-in-depth) vs frontend-only | Frontend-only: spec-compliant (FR-008); Backend-also: robust | Both — frontend rejects empty; backend trims as last defence before comparison |
